# Service Operations V1 — порядок выкладки (EXPAND → ENFORCE)

Миграция намеренно разделена на две фазы, чтобы выкладка была не только логически
корректной, но и **операционно безопасной**. Обе фазы НЕ должны применяться одним
`supabase db push`.

## Почему разделено

`ENFORCE` включает строгий FSM и обязательные поля (`resolution`, причина/дата
переноса, защита терминальных статусов). Если включить его до выкладки нового
фронтенда, **старый `app.service.tsx` начнёт падать** на завершении и продвижении
заявок (старый путь `new → in_progress → done` без `resolution` перестанет
проходить). Поэтому строгие правила включаются отдельным шагом — только после того,
как новый фронтенд развёрнут, проверен, а данные прошли preflight.

## Где что лежит

| Артефакт | Путь | Применяется автоматически? |
|---|---|---|
| EXPAND (миграция) | `supabase/migrations/20260719120000_service_operations_v1_expand.sql` | **Да**, через `supabase db push` |
| EXPAND verify | `supabase/tests/service_operations_v1_expand_verify.sql` | нет (ручной прогон) |
| ENFORCE preflight | `supabase/tests/service_operations_v1_enforce_preflight.sql` | нет (ручной прогон, read-only) |
| ENFORCE (шаблон) | `supabase/deferred/service_operations_v1_enforce.sql` | **Нет** — вне `migrations/`, не попадает под `db push` |
| ENFORCE verify | `supabase/tests/service_operations_v1_enforce_verify.sql` | нет (только после будущей ENFORCE-миграции) |

> ENFORCE хранится в `supabase/deferred/` как **отложенный шаблон**. Он станет
> миграцией позже — отдельным коммитом/PR, скопированным в `supabase/migrations/`
> под **новым актуальным timestamp** (`YYYYMMDDHHMMSS_service_operations_v1_enforce.sql`).
> Старый timestamp заранее НЕ резервируется.

## Порядок выкладки

### A. Применить EXPAND
```
supabase db push
```
Применяется только `…_service_operations_v1_expand.sql` (ENFORCE в `migrations/` нет).
Полностью обратно совместимо: старый фронтенд продолжает работать без изменений.

### B. Прогнать expand_verify
```
psql "$DB_URL" -v ON_ERROR_STOP=1 -f supabase/tests/service_operations_v1_expand_verify.sql
```
Проверяет структуру, RLS, сохранность старых статусов, старый CRUD, завершение без
`resolution`, временные метки, единственную feedback-задачу, отсутствие дублей,
план → первая заявка, done → следующая заявка, изоляцию компаний.

### C. Регенерировать `types.ts`
Обновить сгенерированные типы Supabase под новую схему (новые таблицы/поля), чтобы
фронтенд типизировался против EXPAND-состояния БД.

### D. Развернуть новый фронтенд
Выложить сборку с новым модулем service (доска, диспетчеризация, планы, история,
перезвоны). Новый фронтенд рассчитан на работу **только с EXPAND** и не требует
ENFORCE для запуска (см. раздел «Зависимость фронтенда» ниже).

### E. Проверить CRUD и модули
Проверить вручную на целевой среде:
- старый сценарий заявок (создание/правка/завершение);
- новый CRUD заявок;
- доску (переходы статусов нового FSM);
- очередь перезвонов (callback);
- планы обслуживания (создание плана → первая заявка);
- историю событий (`service_events`).

### F. Прогнать enforce_preflight
```
psql "$DB_URL" -v ON_ERROR_STOP=1 -f supabase/tests/service_operations_v1_enforce_preflight.sql
```
Read-only. Ищет: `done` без `resolution`; неизвестные/устаревшие статусы;
некорректные переходные данные; `rescheduled` без даты/причины; дубли
feedback-задач; дубли следующих заявок; нарушения `company_id` между
`service_requests`, `service_plans`, `tasks`, `service_events`.

**Гейт:** все блокирующие выборки (`preflight_summary`) должны вернуть **0 строк**.
При ненулевом результате preflight падает под `ON_ERROR_STOP`. Preflight ничего не
исправляет — исправление старых данных требует **отдельного разрешения** и делается
до этого шага собственным изменением.

### G. Добавить ENFORCE отдельным PR
Только после зелёного preflight: скопировать `supabase/deferred/service_operations_v1_enforce.sql`
в `supabase/migrations/` под **новым актуальным timestamp** и оформить отдельным
коммитом/PR. Это первый момент, когда ENFORCE попадает в автоприменяемые миграции.

### H. Применить ENFORCE и прогнать enforce_verify
```
supabase db push
psql "$DB_URL" -v ON_ERROR_STOP=1 -f supabase/tests/service_operations_v1_enforce_verify.sql
```
`enforce_verify` подтверждает: допустимые переходы проходят; недопустимые
отклоняются; `done` без `resolution` отклоняется; `rescheduled` без даты/причины
отклоняется; терминальные ограничения работают.

## Зависимость фронтенда от фаз

Новый фронтенд работает **полностью на EXPAND**. ENFORCE — это серверный «ремень
безопасности» для целостности данных, а не предусловие запуска UI:

- переходы статусов, `resolution` при завершении и обязательные поля переноса
  валидируются на стороне фронтенда, а не полагаются на ошибку БД для работы;
- EXPAND уже даёт всю схему (таблицы, поля, индексы, RLS, автоматику), которую
  читает и пишет UI;
- до ENFORCE строгие правила не действуют — фронтенд не ломается и не зависит от
  того, что БД что-то отклонит.

Таким образом порядок B→E можно полностью пройти и эксплуатировать систему на одном
EXPAND; ENFORCE включается позже как усиление инвариантов.

## Запрещено без отдельного разрешения

`push` · `merge` · `PR` · `deploy` · `supabase db push` · применение ENFORCE ·
изменение production · автоисправление старых данных.
